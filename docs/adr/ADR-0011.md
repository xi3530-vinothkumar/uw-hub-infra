# ADR-0011: Multi-repo layout: hub + backend + ai-worker independent git repos

**Status:** Accepted

## Context
The system has three independently deployable layers: a documentation/infra hub
(docker-compose, CLAUDE.md, shared docs), a Java Spring Boot orchestrator, and a Python
FastAPI AI worker. Each layer has a different technology stack, release cadence, and team
ownership. Coupling them in a monorepo would force every CI pipeline to rebuild and
re-test all layers on any commit, and would complicate independent versioning and
deployment.

## Decision
Three separate git repositories:

1. **hub** — docs (`docs/`), docker-compose, CLAUDE.md, `.claude/` rules, and all
   cross-cutting ADRs. Acts as the integration reference. The hub `.gitignore` excludes
   `backend/` and `ai-worker/` directories so those trees are never accidentally committed
   into the hub repo.
2. **backend/** — Java 17 / Spring Boot orchestrator, independently cloneable and buildable
   (`mvn package`).
3. **ai-worker/** — Python 3.11 / FastAPI AI worker, independently cloneable and runnable
   (`uvicorn app.main:app`).

Shared documentation (HLD, LLD, ADRs, KANBAN, PROGRESS) lives in `hub/docs/` and is the
single authoritative source. docker-compose in the hub repo declares the dependency graph
and image references, making the integration contract explicit.

## Consequences
+ Developers clone only the layer they are working on; CI pipelines are scoped per repo.
+ Independent versioning and tagging (e.g. `backend/v1.2.0`, `ai-worker/v0.5.1`).
+ Release cadences are decoupled; a Python hotfix does not require a Java rebuild.
+ Hub docker-compose is the explicit integration contract; all inter-service dependencies
  are declared there.
- Cross-cutting changes (e.g. new queue name, new message schema) must be coordinated
  across repos; no atomic multi-repo commit.
- Contributors must know which repo to clone; onboarding docs in the hub README must be
  clear about the three-repo structure.

## Alternatives considered
- **Monorepo (single git repo):** simpler cross-cutting changes and atomic commits, but
  couples CI pipelines and complicates independent deployment. Rejected because the three
  layers have meaningfully different build tools and release cadences.
- **Monorepo with path-scoped CI triggers:** partially mitigates the CI coupling concern
  but still prevents independent versioning and tagging. Rejected as a half-measure.
