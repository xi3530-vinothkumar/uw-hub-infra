# ADR-0010: Bounded retries (app-level count + TTL + DLQ), prefetch=1

**Status:** Accepted

## Context
Ollama serves one model at a time, so the worker must process one task at a time. A naive
RabbitMQ nack-requeue does not bound retries (infinite loop).

## Decision
Single consumer with `prefetch_count=1`, ack-after-success. On failure: if `retryCount<3`
re-publish (`retryCount+1`) to a TTL retry queue (backoff, dead-letters back to the task
queue); if `≥3` publish to the DLQ. Java's DLQ handler branches on task type: EXTRACT/VISION
→ `FAILED_AI`; NARRATIVE → template fallback + `DECIDED`. Domain write + `task_results`
insert share one transaction (idempotency). Watchdog is activity-based (`last_activity_at`
stale > 15 min).

## Consequences
+ Correct, bounded, backed-off retries; crash-safe; no poison loops; respects single-model Ollama.
- More queues/topology to declare (retry + DLQ per model type).

## Alternatives considered
- Naive nack-requeue: incorrect (infinite retries). Quorum-queue delivery-limit: viable
  alternative; app-level counter chosen for clarity on classic queues.
