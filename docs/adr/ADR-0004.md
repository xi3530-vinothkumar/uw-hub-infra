# ADR-0004: Event-driven pipeline via RabbitMQ

**Status:** Accepted

## Context
Ollama serves one model at a time; each swap is a 15–30s cold load. Synchronous HTTP
Java→Python would block requests and risk timeouts across three sequential AI steps.

## Decision
Java and Python communicate asynchronously through RabbitMQ (task/retry/result/DLQ queues).
Java returns 202 immediately and drives a status pipeline; the frontend polls.

## Consequences
+ Non-blocking UX; model-swap latency absorbed as queue latency; resilient to worker crashes.
+ Natural place for retries, DLQ, and the resilience design.
- Adds broker infrastructure and an orchestration state machine to build (see HLD §9).
- Does NOT add throughput (single Ollama is the bottleneck) — it buys decoupling + resilience.

## Alternatives considered
- Synchronous HTTP: simplest, but blocks and times out on slow model swaps.
- Postgres-as-queue: lighter, but weaker retry/DLQ semantics; RabbitMQ chosen for correctness.
